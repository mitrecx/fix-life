# 每日计划页体验优化 — 设计说明

**日期：** 2026-04-20  
**状态：** 已定稿（待实现）

## 1. 背景与问题

生产环境访问日志显示：用户在「每日计划」中**添加任务**后，前端会触发：

- `POST .../daily-plans/{id}/tasks`（响应快、体积小）；
- 随后 `GET .../daily-plans/?start_date=2020-01-01&end_date=2030-12-31`（约 **115KB+**）。

根因在前端：`DailyPlansList.loadPlans()` 在拉取当周数据后调用 `loadAllPlans()`，固定请求十年区间，用于日期冲突检测。同一页多张 `DailyPlanCard` 各自在 `useEffect` 中调用 `systemSettingsService.getSettings()`，在缓存尚未写入前**并发穿透**，导致多次 `GET /system-settings/me`。

## 2. 目标与非目标

### 2.1 目标

- **跟手（A）**：新增/编辑/删除**日计划任务**后，界面尽快反映结果，减少整表长时间加载感。
- **少大请求（B）**：操作后**不再**常规触发 `2020-01-01`～`2030-12-31` 全量列表请求。
- **冲突处理**：同一用户同一日期不应依赖前端扛全量数据；采用**轻量接口** + **自动合并（Upsert）**（见第 4 节）。

### 2.2 非目标

- 本版不改造「月计划」列表（`MonthlyPlansList`）除非后续单独立项。
- 不强制上虚拟滚动；若实现后仍慢再评估。

## 3. 方案选型

| 方案 | 说明 | 结论 |
|------|------|------|
| 1（采用） | 轻量按日查询 + 创建时 Upsert/合并 + 任务操作只刷当周 + 修复 settings 并发 | **采用** |
| 2 | 批量「已有日期」接口 + 前端复杂缓存 | 暂不采用 |
| 3 | 仅 debounce 全量请求 | 不满足 B |

## 4. 后端设计

### 4.1 按日查询（轻量）

- **用途**：前端日历、提交前提示、「将合并到已有计划」等。
- **建议**：`GET /api/v1/daily-plans/by-date/{plan_date}`（或等价 query 形式），仅当前用户。
- **响应**：该日已有计划时返回 **200** + **最小载荷**（含 `id`、`plan_date`、`title` 等必要字段，**不**默认嵌套全量 `daily_tasks`，除非明确需要）；不存在时 **404**（或统一 200 + `{ exists: false }`，实现阶段二选一并保持全局一致）。

### 4.2 创建日计划 — 自动合并（Upsert）

数据库层面对 `(user_id, plan_date)` 已有唯一约束时，**不能**插入第二行；产品语义为：**再次对同日「创建」→ 合并进已有行**。

- **行为**：`POST /api/v1/daily-plans` 在检测到该日**已存在**计划时：
  - **不新建**记录；
  - 对已有记录按下面规则更新；
  - 响应建议区分 **`201 Created`**（新建）与 **`200 OK`**（已存在并已合并），便于前端可选提示。

**字段合并规则（已定稿）：**

| 字段 | 规则 |
|------|------|
| `title` | **以新为准（A）**：若请求体中 `title` **非空**，则**覆盖**已有标题；若新标题为空，则保留原有标题。 |
| `notes` | 请求体中 `notes` **非空**时，**追加**到已有备注（建议使用明确分隔符，如 `\n---\n`），避免覆盖丢失历史。 |
| `monthly_plan_id` | 仅当**原有值为空**且请求中提供非空值时写入；否则不覆盖已有非空关联。 |

**任务**：当前创建表单不包含批量任务；合并接口**不**负责搬迁任务。若未来表单支持初始任务列表，再扩展「追加任务」及去重策略。

**与 IntegrityError**：若仍存在竞态导致唯一约束冲突，捕获后转为「加载已有 + 合并」或返回明确错误，实现阶段选一种并保持幂等倾向。

## 5. 前端设计

### 5.1 `DailyPlansList`

- `loadPlans()`：**仅**请求当前周 `startDate`～`endDate`。
- **移除**每次 `loadPlans()` 后调用 `loadAllPlans()`（固定十年区间）的路径。
- **删除或收窄** `getAll("2020-01-01","2030-12-31")`：仅保留确需全量的功能（如导出），且须有明确触发条件与注释。
- 任务增删改后的 `onUpdate`：**只刷新当周**（`loadWeekOnly` 语义）；可选第二阶段做乐观更新。

### 5.2 `DailyPlanForm`

- `existingPlans` / `takenDates`：改为结合轻量 **按日接口** 或当周列表 + 按日检查；**禁用「已占用日期」**策略可调整为：选中已占用日期时提示「将合并到已有日计划」或允许选择并由后端 Upsert。
- 提交创建：走统一 `POST`，根据 **200 vs 201** 决定提示文案（可选）。

### 5.3 `system-settings` 风暴修复

根因：多张卡片并发 `getSettings()`，首次请求未完成前缓存为空。

- **必须**：在 `SystemSettingsService` 内对 `getSettings()` 做 **in-flight 去重**（并发共享同一 Promise）。
- **可选**：在 `DailyPlansList` 顶层预取一次 settings，经 props 下发，卡片内不再各自请求。

## 6. 错误与竞态

- 按日检查与提交之间仍可能有竞态；**以后端唯一约束与 Upsert 结果为准**。
- 合并失败或网络错误时按现有 `message`/日志处理；不阻塞用户重试。

## 7. 验收标准

- 在每日计划页执行「添加任务」：网络面板中**不应**在常规操作后出现 `start_date=2020-01-01&end_date=2030-12-31`（除非用户显式触发需全量的功能）。
- 同屏加载时，`GET /api/v1/system-settings/me` **不应**出现与卡片数量相当的并行重复（允许 1 次或 0 次缓存命中）。
- 同一日重复「创建」日计划：**不**再因唯一约束报用户难懂的错误；已有计划被**合并更新**（标题按 **A** 规则，备注追加）。
- 轻量按日接口可用，且查询不默认拖全量任务列表。
